## Warnings (Should Fix)

- [ ] **Stale model IDs in CONVENTIONS.md examples** — `CONVENTIONS.md:444` uses `claude-sonnet-4-20250514` in the `useProviderChat` example, and `CONVENTIONS.md:555` uses the same old ID in the `getModelDisplayName` example with the comment `// -> "Sonnet 4"` (which would now fall through to the raw ID fallback). These should be updated to `claude-sonnet-4-6` and the display name comment to `// -> "Sonnet 4.6"` to match the actual current models.

- [ ] **Stale model IDs in docs/IMPLEMENTATION-PLAN.md** — `docs/IMPLEMENTATION-PLAN.md:209` still lists `GPT-4o, GPT-4o-mini, o1, o3-mini` as OpenAI models. While this is a historical planning document, it could confuse anyone reading the implementation plan to understand the current state.

- [ ] **Stale model IDs in previous review** — `docs/reviews/2026-02-28-phase3-review.md:58` references `claude-sonnet-4-20250514, gpt-4o, gemini-2.0-flash`. The concern noted there (store/settings defaults drifting) was exactly what this PR addresses, but the old review still contains stale IDs in its text. Consider adding a note that this was addressed.

- [ ] **`navigator.platform` is deprecated** — `src/components/TopBar.tsx:50,68` uses `navigator.platform?.includes('Mac')` for platform-aware keyboard shortcut tooltips. `navigator.platform` is deprecated in modern browsers. While it still works and is used only for tooltip text (non-critical), consider using `navigator.userAgentData?.platform` with a fallback, or the simpler heuristic `/(Mac|iPhone|iPad)/.test(navigator.userAgent)`.

- [ ] **Overly permissive `satisfies` type for PROVIDER_OPTIONS** — `functions/api/chat.ts:302` uses `satisfies Record<Provider, Record<string, Record<string, string | boolean | Record<string, string | boolean>>>>`. This type is manually constructed and allows arbitrary nesting, which means it does not actually constrain the shape of `providerOptions` meaningfully. If the AI SDK provides specific types for `providerOptions` (e.g., from `@ai-sdk/anthropic`, `@ai-sdk/openai`, `@ai-sdk/google`), using those would provide compile-time validation that the options are correct. If no SDK types are available, consider at least using `as const satisfies ...` to narrow the literal types.

## Suggestions (Consider)

- [ ] **Extract `DEFAULT_MODELS` as a shared constant** — The same default model IDs (`claude-sonnet-4-6`, `gpt-5.2`, `gemini-3-flash-preview`) are duplicated in three places: `src/lib/store.ts:54-56`, `src/lib/db/settings.ts:20-22`, and `src/lib/models.ts` (first entry of each array). A prior review (`2026-02-28-phase3-review.md:58`) already flagged this drift risk. Consider deriving defaults from `MODEL_OPTIONS` (e.g., `MODEL_OPTIONS[provider][0].id`) in the store and settings to make model updates a single-file change.

- [ ] **Thinking/reasoning should be per-model, not per-provider** — `PROVIDER_OPTIONS` applies reasoning to all models for a given provider. Not all models from a provider may support thinking (e.g., if a future budget model is added). The current 2-model-per-provider setup works fine, but if more models are added later, this structure may need to become model-aware. Worth a comment in the code noting this assumption.

- [ ] **`handleKeyDown` in ConversationSearch was un-memoized** — `src/components/ConversationSearch.tsx` changed `handleKeyDown` from a `useCallback` to an inline function. While this works correctly (the function is only used in a `div.onKeyDown` prop), it means a new function reference on every render. For this component the perf impact is negligible since it is only open during active search. No action needed, but worth noting the intentionality.

- [ ] **Consider testing that `providerOptions` does NOT leak across providers** — The new `providerOptions` tests verify each provider gets its own options, but they do not verify that (for example) the `claude` provider does NOT receive `openai` or `google` keys. Adding a negative assertion like `expect(Object.keys(args.providerOptions!)).toEqual(['anthropic'])` would catch any future regression where options bleed across providers.

- [ ] **`sendReasoning` tests share setup work redundantly** — The three tests in the `sendReasoning flag` describe block each independently call `captureUIStreamOptions()`, setting up and tearing down the same mock. These could share a single `beforeEach` with a shared `opts` variable, reducing test execution time and duplication.

## Patterns to Document

- [ ] **Provider Options pattern**: The `PROVIDER_OPTIONS` constant in `functions/api/chat.ts` introduces a new pattern for configuring per-provider AI SDK options. This should be documented in CONVENTIONS.md under the "Cloudflare Pages Functions" section, noting that provider-specific options (thinking, reasoning, etc.) are defined as a static map keyed by provider and passed to `streamText` via `providerOptions`.

- [ ] **`sendReasoning` flag**: The addition of `sendReasoning: true` in `toUIMessageStreamResponse` is a new pattern that enables reasoning content to flow through the stream to the client. This should be documented alongside the existing "Token Usage Extraction Pipeline" convention, noting that reasoning tokens are included in the stream when thinking is enabled.
